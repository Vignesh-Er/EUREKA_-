## WHERE TO PUT THE ORCHESTRATION CODE

The orchestration logic should live in the backend, not in the frontend and not inside individual engine routers.

Recommended placement:
- Main workflow coordinator: `services/api/app/services/workflows/orchestrator.py`
- Workflow definitions / enums: `services/api/app/services/workflows/definitions.py`
- Cross-engine handoff schemas: `services/api/app/schemas/handoffs.py`
- Queue / retry jobs: `services/api/app/services/jobs/workflow_jobs.py`
- Event publishing helpers: `services/api/app/services/events/`
- Observability / audit logging: `services/api/app/services/monitoring/`

## SEQUENTIAL ENGINE WIRING ORDER

Integrate the engines in this exact order:

1. Build and verify PostgreSQL, Qdrant, Redis, object storage, auth, and FastAPI base app
2. Wire Engine 1 (`discovery_service.py`) and confirm it returns valid JSON
3. Wire Engine 3 (`journey_service.py`) to consume Engine 1 handoff output
4. Wire Engine 2 (`context_service.py`) to generate topic Context Cards from Engine 3 outputs
5. Wire Engine 5 (`companion_service.py`) for professor uploads and lecture companion generation
6. Wire Engine 4 (`evaluation_service.py`) to consume lecture topics and return assessments
7. Wire Engine 7 (`translation_service.py`) to translate outputs from Engines 1, 2, 3, 4, and 5
8. Wire Engine 6 (`admin_service.py`) for operational workflows
9. Only after all engines work independently, implement `orchestrator.py` to manage handoffs, retries, and workflow state
10. Build complete Next.js frontend with design system, all role-specific screens, and gamification (see Phase 6 in 00_Implementation_Strategy.md)
11. Connect `apps/web` to backend APIs and wire real-time event architecture after orchestration is stable

## BEGINNER-FRIENDLY ORCHESTRATION RULE

Do **not** build orchestration first.

As a beginner, you should only start orchestration after these are already true:
- Engine 1 works alone
- Engine 2 works alone
- Engine 3 works alone
- Engine 4 works alone
- Engine 5 works alone
- Engine 6 works alone
- Engine 7 works alone

If you try orchestration too early, you will not know whether a failure came from the router, the service, the model call, the schema, or the handoff.

### SSE Event Persistence & Delivery Guarantees

SSE connections can drop (tab backgrounded, network change, Vercel cold start). Events sent while the client is disconnected are lost unless persisted server-side.

**Server-Side Event Store:**
- All SSE events are written to a Redis Stream (`eureka:events:{user_id}`) with a TTL of 24 hours
- Each event entry: `{event_id, event_type, payload, created_at, delivered: false}`
- When a client reconnects, it sends `Last-Event-ID` header → server replays all events after that ID from the Redis Stream
- Events older than 24 hours are auto-trimmed (`XTRIM MAXLEN ~1000`)

**Delivery Flow:**
```
1. Backend publishes event → Redis XADD eureka:events:{user_id}
2. SSE endpoint reads from Redis XREAD (blocking) → pushes to client
3. Client receives event → stores Last-Event-ID locally
4. On disconnect + reconnect → client sends Last-Event-ID
5. Server replays missed events from Redis Stream (XRANGE)
6. Normal streaming resumes
```

**Vercel / Serverless Considerations:**
- Vercel serverless functions have a **30-second timeout** on Hobby and **300-second timeout** on Pro — SSE connections will drop
- **Recommended approach:** Use Vercel's Edge Runtime (`export const runtime = 'edge'`) for the SSE endpoint — no timeout limit
- Alternative: Use an external service (Upstash Redis with SSE, or a dedicated WebSocket server on Railway/Fly.io)
- The Socket.IO WebSocket channel requires a persistent server — **cannot run on Vercel serverless**. Options: (a) deploy WebSocket server separately on Railway/Fly.io, (b) replace Socket.IO with polling + SSE for processing status

Optimistic Updates:
  • Assessment submission → Immediately show "Submitted ✓" → Confirm on server response
  • Language switch → Show skeleton → Replace content on translation arrival
  • Feedback submission → Toast "Submitted anonymously" → No server confirmation shown
  • Profile settings → Immediate UI update → Revert on server error

Error Handling:
  • Network errors → Toast with retry button + offline indicator in header
  • AI timeout (>120s) → "Taking longer than expected..." message + auto-retry once
  • 401 Unauthorized → Silent token refresh → Retry original request
  • 403 Forbidden → Redirect to appropriate role dashboard
  • 500 Server Error → Error boundary with "Something went wrong" + report button
